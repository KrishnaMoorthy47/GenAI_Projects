# CodeSentinel

AI-powered GitHub PR reviewer that detects security vulnerabilities and code quality issues. Triggered via **GitHub webhook** (real-time) or **REST API** (manual), then posts a structured Markdown review directly to the PR.

## Architecture

```
GitHub Webhook (PR opened/updated)    POST /review (manual API)
              │                                  │
              ▼                                  ▼
   verify HMAC signature              verify API key
              │                                  │
              └────────────┬─────────────────────┘
                           ▼
                      parse_diff
                     /           \
          (parallel superstep)    \
         /                         \
  security_agent              quality_agent
  • OWASP regex scan          • Complexity analysis
  • LLM vulnerability review  • Error handling review
         \                         /
          \                       /
           ▼                     ▼
           merge_findings (barrier)
                    │
                    ▼
             summary_agent
          (final Markdown review)
                    │
                    ▼
       post_pr_review → GitHub PR Comment
```

**Parallelism:** `security_agent` and `quality_agent` run as a true parallel superstep — LangGraph fans out and waits for both before `merge_findings`.

## Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/review` | Manually trigger a PR review |
| `GET` | `/review/{id}` | Get review status and results |
| `POST` | `/webhook/github` | GitHub webhook receiver (HMAC verified) |
| `GET` | `/health` | Health check |

## Quick Start

### With Docker

```bash
cp .env.example .env
# Fill in OPENAI_API_KEY (or Azure), GITHUB_TOKEN, GITHUB_WEBHOOK_SECRET

docker compose -f docker/docker-compose.yml up --build
```

### Local development

```bash
uv venv && source .venv/bin/activate
uv pip install -r requirements.txt
uvicorn codesentinel.main:app --reload --port 8001
```

## Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `API_KEY` | Yes | `x-api-key` header auth value |
| `LLM_PROVIDER` | Yes | `openai` or `azure_openai` |
| `OPENAI_API_KEY` | If OpenAI | OpenAI API key |
| `OPENAI_MODEL` | No | Model (default: `gpt-4o`) |
| `AZURE_OPENAI_ENDPOINT` | If Azure | Azure OpenAI endpoint |
| `AZURE_OPENAI_DEPLOYMENT` | If Azure | Deployment name |
| `AZURE_OPENAI_API_KEY` | If Azure | Azure key |
| `AZURE_OPENAI_API_VERSION` | If Azure | API version |
| `GITHUB_TOKEN` | Yes | GitHub PAT with `pull_requests` write scope |
| `GITHUB_WEBHOOK_SECRET` | For webhooks | Shared secret set in GitHub repo settings |
| `DATABASE_URL` | No | SQLite path (default: `sqlite+aiosqlite:///./codesentinel.db`) |

## Example Usage

### Manual API trigger

```bash
curl -X POST http://localhost:8001/review \
  -H "x-api-key: dev-secret" \
  -H "Content-Type: application/json" \
  -d '{
    "repo": "your-username/test-repo",
    "pr_number": 1,
    "github_token": "ghp_..."
  }'
# → {"review_id": "abc-123", "status": "pending", ...}

# Poll for results
curl http://localhost:8001/review/abc-123 \
  -H "x-api-key: dev-secret"
```

### GitHub Webhook setup

1. Go to your repo → Settings → Webhooks → Add webhook
2. Payload URL: `https://your-domain.com/webhook/github` (use [ngrok](https://ngrok.com) for local testing)
3. Content type: `application/json`
4. Secret: same value as `GITHUB_WEBHOOK_SECRET`
5. Events: select **Pull requests**

### Webhook local testing with ngrok

```bash
ngrok http 8001
# Copy the forwarding URL and set it as webhook URL in GitHub
```

## Example Review Output

```markdown
## CodeSentinel Review

### Summary
This PR introduces a user authentication endpoint with critical security vulnerabilities
that must be addressed before merging.

### Security Findings
- 🔴 **[CRITICAL] SQL Injection** (`app/api/users.py:7`)
  `cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")` — user input
  directly interpolated into SQL query. Use parameterized queries instead.

- 🔴 **[CRITICAL] Hardcoded Credential** (`app/api/users.py:11`)
  `db_password = "hardcoded_secret_123"` — secrets must not be in source code.
  Use environment variables or a secrets manager.

### Code Quality
- 🟡 **[MEDIUM]** Missing error handling for database connection failures
- 🟡 **[MEDIUM]** Function `get_user` has no return type annotation

### Positive Observations
- Good separation of authentication logic into dedicated function

### Recommendation
**REQUEST_CHANGES** — Critical security vulnerabilities (SQL injection, hardcoded credentials) must be fixed.

---
*🤖 Generated by CodeSentinel — AI-powered PR review*
```

## OWASP Patterns Detected

CodeSentinel includes static regex-based detection for:

| ID | Category | Severity |
|----|----------|----------|
| OWASP-A01-SQL-001 | SQL Injection | Critical |
| OWASP-A01-CMD-001 | Command Injection | Critical |
| OWASP-A02-HARDCODED-SECRET-001 | Hardcoded Secrets | Critical |
| OWASP-A02-WEAK-HASH-001 | Weak Cryptography | High |
| OWASP-A03-XSS-001 | Cross-Site Scripting | High |
| OWASP-A05-DEBUG-001 | Debug Mode Enabled | Medium |
| OWASP-A05-CORS-001 | CORS Wildcard | Medium |
| OWASP-A07-AUTH-001 | SSL Verification Disabled | High |
| OWASP-A09-LOGGING-001 | Sensitive Data in Logs | Low |

Plus LLM-powered semantic analysis for context-aware vulnerability detection.

## Tech Stack

- **[LangGraph](https://github.com/langchain-ai/langgraph)** — Parallel agent superstep (fan-out/fan-in)
- **[FastAPI](https://fastapi.tiangolo.com)** — Async REST API + webhook receiver
- **[PyGithub](https://pygithub.readthedocs.io)** — GitHub API integration
- **[SQLAlchemy](https://www.sqlalchemy.org) + aiosqlite** — Async review history persistence
- **stdlib `hmac`** — Webhook signature verification (no extra package needed)
